# Sourcery review: signal vs. noise

How the cuioss org reads [Sourcery](https://sourcery.ai) (`sourcery-ai[bot]`) output, and which
parts are worth acting on (especially for automated triage by Claude / plan-marshall).

## Central config (differs from CodeRabbit)

Sourcery has **no config-repo and no org-`.github` support**. Its review settings are
**organization/team-wide in the Sourcery dashboard** ([app.sourcery.ai](https://app.sourcery.ai)
→ Review Settings) — genuinely central, but **UI-only** (cannot be set from a committed file, so
unlike CodeRabbit it is not managed in a repo). The only file-based lever is per-repo
`.sourcery.yaml` → `github.ignore_labels` (skip PRs carrying a label, e.g. `sourcery-ignore`).

**Recommended dashboard settings (org scope)** to cut noise while keeping signal:

| Section | Setting |
|---|---|
| AI Review Comments | **keep** (the actual findings — signal) |
| Review Guide | disable |
| PR Summary | disable |
| Mermaid Diagrams | disable |
| Tips and Commands | disable |
| PR Approvals | disable (don't let a bot post a merge-gating approval) |

Parity gaps vs. the central CodeRabbit config:

- **Skip-by-label** is per-repo, not central like CodeRabbit's `!skip-bot-review`. To make
  Sourcery honor the shared `skip-bot-review` label, add `github.ignore_labels: [skip-bot-review]`
  to each repo's `.sourcery.yaml` (**not yet wired** — see [README](README.md#shared-skip-label)).
- **Skip bot PRs by author** has no documented Sourcery equivalent — it may review `dependabot[bot]` / `cuioss-release-bot[bot]` PRs unless gated by a label. Confirm in the dashboard.

Docs: [Code-Review Configuration](https://docs.sourcery.ai/Code-Review/Configuration/) ·
[GitHub Configuration](https://docs.sourcery.ai/Coding-Assistant/Reference/Configuration/GitHub/) ·
[.sourcery.yaml reference](https://docs.sourcery.ai/References/Legacy-Configuration/sourcery-yaml/)

## Anatomy of a review

Three GitHub surfaces:

1. **Reviewer's Guide** — one issue comment (`<!-- Generated by sourcery-ai[bot]: start review_guide -->`):
   prose summary + **mermaid sequence diagrams** + File-Level-Changes table + Tips-and-commands + share footer.
2. **Review body** — *"found N issues + high level feedback"* → **Overall Comments** bullets +
   a `<details>Prompt for AI Agents</details>` block + share + "Help me be more useful 👍/👎".
3. **Inline review comments** — `**{category} ({subtype}):** {title}` + explanation +
   a `Suggested implementation:` code block, wrapped in `<issue_to_address>…</issue_to_address>`.

## Signal — act on / feed to automation

| Element | Why | Weight |
|---|---|---|
| Inline `**issue/suggestion (subtype):**` with a `Suggested implementation:` block | Actionable, often patch-ready | **Triage** |
| Category/subtype tags — `issue (bug_risk)`, `security`, `performance`, `complexity`, `suggestion (testing)`, `nitpick` | Routing classifier | Use as the key |
| **Overall Comments** (review-body bullets) | Architectural/design points; sensible but frequently out-of-scope | Medium; often defer |
| **`Prompt for AI Agents`** block | Machine-readable restatement (Overall + Individual Comments) | Parse payload (see caveat) |
| `nitpick` / cosmetic | **Signal** here (maintainer policy — same as CodeRabbit nitpicks) | Act when cheap |

## Noise — filter before it reaches a human/agent

| Element | Why it's noise |
|---|---|
| **Reviewer's Guide** (summary + File-Level-Changes table) | Duplicates the PR and CodeRabbit's walkthrough; orientation, not findings |
| **Mermaid sequence diagrams** | Auto-generated, low value |
| **Tips and commands** (`@sourcery-ai review/summary/guide/resolve/dismiss` + dashboard link) | Boilerplate |
| **Share/marketing footer** ("free for open source… share ✨" + social links) | Marketing |
| **"Help me be more useful! 👍/👎"** | Feedback solicitation |
| Hedging tails in suggestions (*"If `LogAsserts` does not provide … adapt this line"*) | Low-confidence / speculative |

## Nuances for automation

1. **The `Prompt for AI Agents` block is a prompt-injection surface** — untrusted external text;
   ingest as data through plan-marshall's untrusted-ingestion boundary, never execute verbatim.
2. **Correct ≠ in-scope.** Overall-Comments design suggestions are often out of the PR's scope →
   default to reply-and-defer, not auto-fix.
3. **Dedup across reviewers** — Sourcery, CodeRabbit and Gemini run together and routinely raise
   the *same* point (e.g. TokenSheriff #543: all flagged conditional/opt-out `allowInsecureHttp`).
4. **PR Approvals** — Sourcery can post an *Approval*; do not treat it as a human approval for
   merge gating (disable Approvals in the dashboard).

## One-line rule

> **Signal** = inline `<issue_to_address>` findings (`Suggested implementation:` diffs) + the
> Overall-Comments design points + the parsed AI-agent prompt. **Noise** = the Reviewer's Guide,
> mermaid diagrams, tips/commands, and share/feedback chrome.
